Every error ‚Äî from auth failures to validation errors to 429s ‚Äî follows the same JSON shape. Client code can pattern-match on `error_code` rather than parsing HTTP status alone.

## Canonical shape

```json
{
  "error": "short_error_slug",
  "error_code": "specific_machine_readable_code",
  "message": "Human-readable description of what went wrong.",
  "details": {
    "field": "optional, present on validation errors"
  }
}
```

- `error` ‚Äî short category slug (`unauthorized`, `not_found`, `validation_error`, `rate_limit_exceeded`, `server_error`).
- `error_code` ‚Äî machine-readable code for programmatic handling. Stable across versions.
- `message` ‚Äî human-readable description. Safe to show to developers; not intended for end users.
- `details` ‚Äî optional context. Present on validation errors (offending field, received value) and rate-limit errors (limit, reset time).

## Status codes

| Status | Category | Common causes |
|---|---|---|
| 400 | Validation error | Invalid query parameter, malformed date, unknown ticker, `per_page` out of range |
| 401 | Unauthorized | Missing API key, invalid key, revoked key |
| 403 | Forbidden | Account suspended, key disabled |
| 404 | Not found | No data for the requested ticker / CBSA / FIPS / industry code |
| 422 | Unprocessable entity | Semantic validation failure (e.g., date range outside covered period) |
| 429 | Rate limited | Daily or burst cap exceeded ‚Äî see [Rate limits](/getting-started/rate-limits) |
| 500 | Server error | Something broke on our side. Transient ‚Äî retry with backoff. Report persistent 500s to [support@thesma.dev](mailto:support@thesma.dev) with the response body. |
| 503 | Service unavailable | Upstream data source (SEC, Census, BLS, SBA) is temporarily unreachable. Transient. |

## Common `error_code` values

### Auth (401 / 403)

| `error_code` | Fix |
|---|---|
| `missing_api_key` | Pass the key via `X-API-Key` or `Authorization: Bearer` header |
| `invalid_api_key` | Key does not exist or has been revoked. Generate a new key in the portal. |
| `account_suspended` | Contact support ‚Äî account is suspended |

### Validation (400 / 422)

| `error_code` | Fix |
|---|---|
| `invalid_parameter` | Check the `details.field` ‚Äî parameter value is malformed or out of range |
| `unknown_ticker` | Ticker not in the SEC EDGAR universe. See [SEC EDGAR coverage](/datasets/sec-edgar). |
| `unknown_cbsa` | CBSA code not recognised. See [US Census coverage](/datasets/us-census). |
| `date_out_of_range` | Requested date is outside the dataset's covered period |

### Rate limits (429)

| `error_code` | Fix |
|---|---|
| `daily_limit_exceeded` | Wait until midnight UTC or upgrade tier. See [Rate limits](/getting-started/rate-limits). |
| `burst_limit_exceeded` | Wait 1 second and retry. Usually means you're paginating too fast. |

### Server (500 / 503)

| `error_code` | Fix |
|---|---|
| `internal_error` | Retry with exponential backoff. Report persistent cases. |
| `upstream_unavailable` | Upstream data source is down. Retry later. |

## Error envelope, always

Even on 500s the response is JSON with the above shape ‚Äî the API never returns a bare HTML error page. If you get one, it's from a layer **in front** of the API (your proxy, a load balancer, a caching CDN) and worth investigating there.

## Retries

- **4xx** are your problem ‚Äî don't retry unless the cause is transient (e.g., 429).
- **5xx** are our problem ‚Äî retry with exponential backoff (the Python SDK does this automatically).
- **429** ‚Äî respect `Retry-After` and retry after the specified number of seconds.
